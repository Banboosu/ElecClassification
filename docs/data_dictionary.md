# 数据字段与标签约定

## 字段

| 字段 | 当前用途 | 说明 |
|---|---|---|
| `charging_station_id` | 样本唯一 ID | 当前文件中每行均唯一，不能直接作为设备分组字段。 |
| `InsertedColumn` | 分类标签 | 训练前转为字符串，再由 `LabelEncoder` 编码。 |
| `charging_powers_str` | 功率时间序列 | 逗号分隔的浮点数序列。 |

当前 CSV 没有设备 ID、站点分组字段或时间戳，因此暂时不能完成按设备或时间的外推评估。
如果原始系统能提供这些字段，应优先补充，而不是从唯一记录 ID 中猜测设备关系。

## 标签

项目旧代码使用以下业务约定：

| 标签 | 项目中的含义 | 是否进入当前三分类实验 |
|---|---|---|
| `0` | 正常 | 是 |
| `1` | 充电器故障 | 是 |
| `2` | 电池异常 | 是 |
| `5` | 无效或不完整记录 | 否 |

标签 `0`、`1`、`2` 的名称来自 `legacy/ceshi.py` 中原有诊断输出。标签 `5` 的“不完整”定义
来自当前项目配置和既有 README；其数据也呈现明显的短序列特征：11,435 条标签 5 中绝大多数
短于 18 个时间点。该统计支持将其从故障类别中分离，但不能代替业务标注依据。

论文定稿前仍需由数据提供者确认并补充：

1. 原始数据采集系统和采样间隔；
2. 标签由人工、规则还是设备告警生成；
3. 三种有效标签的判定标准；
4. 标签 5 的产生条件，以及是否可能包含真实故障；
5. 数据脱敏方式和允许在论文中披露的范围。

在这些信息确认前，论文应使用“项目数据中的标签约定”，不要表述成已经获得外部权威验证的
行业标准。
